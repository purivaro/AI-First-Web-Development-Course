## 🎨 ส่วนที่ 2 — Design System: ภาษาของนักออกแบบมืออาชีพ
*(20 นาที)*

### 2.1 ทำไมเว็บมืออาชีพถึงดู "สะอาด"?
ลองเปรียบเทียบ:
- **เว็บมือสมัครเล่น:** สีแต่ละหน้าไม่เหมือนกัน, ฟอนต์ปนกัน 3-4 แบบ, ระยะห่างไม่สม่ำเสมอ
- **เว็บระดับโปร (Apple, Stripe, Notion):** ทุกหน้า "รู้สึก" เป็นเว็บเดียวกัน เพราะมี **Design System**

> **Design System = ชุดกฎและส่วนประกอบที่ใช้ซ้ำได้ทั่วทั้งเว็บ** — เปรียบเหมือน "Brand Guideline" ของบริษัท

### 2.2 4 องค์ประกอบหลักของ Design System
| องค์ประกอบ | คืออะไร | ตัวอย่าง |
|---|---|---|
| **Color Palette** | ชุดสีหลักที่ใช้ทั่วเว็บ | Primary (สีแบรนด์), Surface (พื้นหลังการ์ด), Background, Text |
| **Typography Scale** | ระบบขนาดและน้ำหนักตัวอักษร | H1 (48px), H2 (32px), Body (16px), Caption (12px) |
| **Spacing Scale** | ระบบระยะห่างที่เป็นมาตรฐาน | 4, 8, 16, 24, 32, 48, 64 px |
| **Components** | ส่วนประกอบสำเร็จที่นำไปใช้ซ้ำ | Button, Card, Input, Navbar |

### 2.3 Design Tokens — ตัวแปรของ Design System
**Design Tokens** = ชื่อสั้นๆ ที่แทนค่าจริง เช่น:
- `--color-primary` → `#3B82F6`
- `--spacing-md` → `16px`
- `--font-heading` → `Inter, sans-serif`

**ข้อดี:** ถ้าอยากเปลี่ยนสีหลักของเว็บทั้งหมด — แก้ที่ Token ที่เดียว → เปลี่ยนทั้งเว็บอัตโนมัติ

### 2.4 ทำไมต้องรู้เรื่องนี้ก่อนสั่ง AI?
เพราะ Prompt ที่ดีจะระบุ Design System ของเรา → ผลลัพธ์ที่ AI สร้างมาจะสอดคล้องกันทั้งเว็บ

❌ *"ทำหน้านี้ให้สวยหน่อย"*
✅ *"ทำหน้านี้โดยใช้ Design System: สีหลัก `#3B82F6`, ฟอนต์ Inter, spacing สเกล 4-8-16-24-32, มุมโค้ง `rounded-lg`"*

> 🎯 **Key Takeaway:** เรา = สถาปนิกที่ออกแบบ "ระบบ", AI = ผู้รับเหมาที่สร้างตามระบบ

## ⚡ ส่วนที่ 3 — พระเอกของงาน: Google Stitch + MCP
*(30 นาที)*

### 3.1 Google Stitch คืออะไร
เครื่องมือออกแบบ UI ด้วย AI จาก Google ที่ต่างจาก ChatGPT/Claude ทั่วไปตรงที่ **มีหน้าจอ Visual ให้คลิกปรับแต่งได้** ก่อนจะได้โค้ด — เสมือนมี Designer ส่วนตัวที่เก่งทั้ง Figma และ Code

### 3.2 Step-by-Step การใช้ Stitch
1. **เข้าเว็บ Google Stitch** และล็อคอินด้วย Google Account
2. **ใส่ Prompt:** copy เนื้อหาจาก `requirements.md` ของ Session 1 มาแปะในช่อง Prompt
3. **ตั้งค่า Design System** ใน Stitch:
   - **Color Palette:** เลือก Primary / Surface / Background
   - **Typography:** เลือกฟอนต์หลัก
   - **Style direction:** Modern / Playful / Minimal / Bold
4. **Micro-adjustments:** คลิกที่ส่วนต่างๆ ของ UI แล้วสั่งสั้นๆ
   - *"เปลี่ยนรูปนี้ให้กลม"*
   - *"เพิ่มพื้นที่ว่างตรงนี้ให้ดูมินิมอลขึ้น"*
   - *"ทำการ์ดให้มีเงาอ่อนๆ"*
5. **Export to Code:** กดปุ่มให้ Stitch คายโค้ดออกมาเป็น HTML + Tailwind CSS (หรือ React)

### 3.3 เชื่อมต่อพลังด้วย MCP (Model Context Protocol)
**ปัญหา:** ทุกครั้งที่แก้ดีไซน์ใน Stitch ต้อง Export → copy → paste เข้า Editor → ยุ่งยากและไม่ sync

**MCP คือทางออก:** ให้ AI ฝั่ง Design (Stitch) และฝั่ง Code (Antigravity) **"คุยกันเอง"** ได้โดยตรง

**กลไก:**
1. **Stitch = MCP Server** — เปิด endpoint ให้ AI ตัวอื่นเข้ามาอ่าน Design System ได้
2. **Antigravity = MCP Client** — เชื่อมไปยัง Stitch ดึงข้อมูลล่าสุดได้ตลอด
3. **Real-time Sync:** เปลี่ยนสีใน Stitch → AI ใน Antigravity ตรวจจับและเสนออัปเดต CSS/Tokens ในโค้ดทันที

**ตัวอย่างคำสั่งที่ทรงพลัง:**
> *"ดึงดีไซน์หน้า Hero ล่าสุดจาก Stitch มาอัปเดตในโปรเจกต์ของเรา ใช้ Design Tokens ที่ Stitch กำหนดไว้"*

## 🔍 ส่วนที่ 4 — ส่องโค้ดที่ Stitch ส่งออกมา
*(15 นาที)*

### 4.1 เปิดโค้ดที่ได้
นำโค้ดที่ Export จาก Stitch มาวางใน Antigravity แล้วเปิดดู

### 4.2 ชี้ให้ดูว่าอะไรเป็นอะไร (Quick Tour)
*ไม่ต้องจำ — แค่ "เห็น" ว่าอะไรอยู่ตรงไหน*

**ในไฟล์ HTML:**
- `<div>`, `<section>` = กล่องและโซน
- `class="..."` = ที่ Tailwind class อยู่

**ใน class ของ Tailwind ที่จะเจอบ่อย:**
| สิ่งที่เห็น | คือ |
|---|---|
| `bg-blue-500`, `text-white` | **สี** (background, text) |
| `p-4`, `m-2`, `px-6`, `py-3` | **Padding / Margin** (ระยะห่าง) |
| `flex`, `grid`, `gap-4` | **Layout** (ระบบจัดวาง) |
| `text-xl`, `font-bold` | **Typography** (ขนาด/น้ำหนักตัวอักษร) |
| `rounded-lg`, `shadow-md` | **Effect** (มุมโค้ง, เงา) |
| `md:flex`, `lg:text-2xl` | **Responsive** (เปลี่ยนตามขนาดจอ) |

### 4.3 Padding vs Margin — ภาพเดียวจบ
```
┌─────────────────────────┐  ← Margin (ระยะนอกกล่อง)
│  ┌───────────────────┐  │
│  │  Padding          │  │  ← Padding (ระยะในกล่อง)
│  │   ┌───────────┐   │  │
│  │   │  Content  │   │  │
│  │   └───────────┘   │  │
│  └───────────────────┘  │
└─────────────────────────┘
```

### 4.4 เรียนรู้ต่อด้วยตัวเอง — ถาม AI เลย!
**Pattern ที่อยากให้ติดตัว:**
- เจอ class ไม่รู้จัก → ไฮไลต์ → ถาม AI: *"บรรทัดนี้คืออะไร ทำหน้าที่อะไรบนหน้าเว็บ?"*
- อยากเปลี่ยน → ถาม AI: *"ถ้าอยากให้กล่องนี้มีระยะห่างจากขอบจอมากขึ้น ต้องแก้ตรงไหน?"*

> 💡 **นี่คือ AI-First Mindset:** เราไม่ต้องเรียน CSS หลายเดือนก่อน — แค่รู้ว่าอะไรเรียกชื่อว่าอะไร แล้วถาม AI ตอนที่ต้องการ

## 💻 ส่วนที่ 6 — JavaScript เบื้องต้น
*(30 นาที)*

### 6.1 JavaScript คืออะไร?
ถ้าเปรียบเทียบกับร่างกาย:
- **HTML** = โครงกระดูก
- **CSS** = ผิวหนัง/เสื้อผ้า
- **JavaScript** = กล้ามเนื้อ + สมอง (ขยับได้ คิดได้ ตอบสนองได้)

**JS ทำอะไรได้บนเว็บ:**
- ฟังเหตุการณ์ (Event) — เช่น "ผู้ใช้คลิกปุ่ม"
- เปลี่ยนหน้าเว็บแบบ Real-time โดยไม่ต้อง reload
- เก็บข้อมูลชั่วคราว (State) ในหน่วยความจำ
- คุยกับ Server / Database ผ่าน API

### 6.2 ตัวอย่างเบื้องต้น (อ่านพอ ไม่ต้องเขียน)
```javascript
// ฟังว่าปุ่มถูกคลิก แล้วทำอะไร
button.addEventListener('click', () => {
  alert('สวัสดี!');
});

// เก็บค่าในตัวแปร
let count = 0;
count = count + 1;  // กลายเป็น 1
```

### 6.3 คำศัพท์ที่ต้องเรียกชื่อให้ถูก
| คำศัพท์ | ความหมายแบบบ้านๆ |
|---|---|
| **Variable (ตัวแปร)** | กล่องเก็บค่า เช่น `count`, `username` |
| **Function (ฟังก์ชัน)** | ชุดคำสั่งที่ทำซ้ำได้ เช่น `addToCart()` |
| **Event (เหตุการณ์)** | การกระทำของผู้ใช้ เช่น click, submit, scroll |
| **State (สถานะ)** | ข้อมูลปัจจุบันที่หน้าเว็บจำได้ขณะใช้งาน |
| **DOM** | โครงสร้างหน้าเว็บที่ JS เข้าไปแก้ไขได้ |
| **API** | ช่องทางคุยกับโลกภายนอก (Server, Database) |

> 💡 **หน้าที่ของเรา:** ไม่ใช่จำ syntax แต่คือ "เรียกชื่อ" ให้ถูก เพื่อสั่ง AI ได้แม่น

### 6.4 ลองสั่ง AI ใส่ JS ในเว็บของเรา
ในเว็บที่เพิ่งทำเสร็จจาก Stitch ลองสั่ง AI:
> *"ช่วยเพิ่มปุ่มในหน้านี้ ตอนคลิกแล้วให้ข้อความเปลี่ยนจาก 'Hello' เป็น 'สวัสดีครับ' ใช้ JavaScript ธรรมดา (vanilla JS) ไม่ต้องใช้ framework"*

ดูผลลัพธ์ — AI จะใส่ `<button>` + `addEventListener` + เปลี่ยน DOM ให้
- **ไฮไลต์โค้ดที่ AI เขียน → ถาม:** *"บรรทัดนี้ทำอะไร?"* → ได้คำอธิบายทันที

> 📌 **ในคาบหน้า (Session 3)** เราจะรู้จัก **React** — Framework ที่ AI ถนัดที่สุด สำหรับสร้าง Web App ขนาดใหญ่ที่มี State และ Database

## 🐛 ส่วนที่ 7 — AI-Assisted Debugging
*(40 นาที)*

### 7.1 ทำไม Debug จึงสำคัญในยุค AI?
AI เขียนโค้ดเก่ง แต่ไม่ได้แปลว่าโค้ดจะทำงานถูกต้อง 100% — บัค (Bug) ยังเกิดได้เสมอ เช่น:
- คลิกปุ่มแล้วไม่มีอะไรเกิดขึ้น
- หน้าจอขาวเปล่า
- มี error สีแดงขึ้นใน Console
- รูปไม่โหลด, ลิงก์ไม่ทำงาน

**ทักษะ AI-First Architect:** ไม่ใช่เขียนโค้ดให้บัคน้อย — แต่คือ **debug ได้เร็วโดยใช้ AI เป็นเครื่องมือ**

### 7.2 รู้จัก DevTools — กล่องเครื่องมือของนักพัฒนา
เปิดด้วย **คลิกขวา → Inspect** (หรือ `F12` / `Cmd+Option+I`)

**4 แท็บที่ใช้บ่อย:**
| แท็บ | หน้าที่ | ใช้ตอนไหน |
|---|---|---|
| **Elements** | ดูโครงสร้าง HTML + CSS ของแต่ละ element | หน้าตาเพี้ยน, อยากรู้ว่าอะไรอยู่ตรงไหน |
| **Console** | แสดง error และ log จาก JavaScript | คลิกแล้วไม่ทำงาน, มีบัค |
| **Network** | ดูว่าเบราว์เซอร์โหลดอะไรจาก Server บ้าง | รูปไม่โหลด, API ไม่ทำงาน |
| **Application** | ดู localStorage / Cookies | เช็คข้อมูลที่เก็บในเครื่อง |

### 7.3 The Debug Workflow — สูตร 3 ขั้นแก้บัคด้วย AI

#### ขั้นที่ 1: เปิด Console ดู Error
- เปิด DevTools → แท็บ **Console**
- ถ้ามี error สีแดง → **copy ข้อความทั้งหมด** (รวมถึงชื่อไฟล์ + เลขบรรทัด)

#### ขั้นที่ 2: ส่งให้ AI พร้อมบริบท
**Pattern คำสั่ง:**
> *"เว็บของฉันคลิกปุ่มแล้วไม่ทำงาน เปิด Console แล้วเจอ error นี้:*
>
> *```*
> *Uncaught TypeError: Cannot read property 'addEventListener' of null*
> *    at app.js:15*
> *```*
>
> *ช่วยอธิบายว่าเกิดอะไรขึ้น และแก้ยังไง"*

#### ขั้นที่ 3: Screenshot ก็ทำได้
ถ้าเป็นบัค **ภาพ** (เช่น layout เพี้ยน, สีผิด):
1. กด `Cmd+Shift+4` (Mac) / `Win+Shift+S` (Windows) จับภาพ
2. แปะลงในแชท AI พร้อมคำสั่ง:
   > *"หน้านี้ของฉันเลย์เอาต์เพี้ยน การ์ดมันยืดเต็มจอ ดูจากภาพแล้วช่วยบอกว่าน่าจะเกิดจาก class ไหน และแก้ยังไง"*

### 7.4 Workshop: ฝึก Debug จริง (ทำในห้อง)
**ภารกิจ:**
1. ในเว็บของตัวเอง ให้สั่ง AI:
   > *"ช่วยใส่บัคแบบสุ่มลงในโค้ดให้ฉัน 1 ตัว (เช่น สะกดชื่อ class ผิด, ลืม event listener) เพื่อให้ฉันฝึก debug — อย่าบอกฉันว่าใส่ตรงไหน"*
2. รันเว็บ → เปิด DevTools → หา error
3. ใช้ **Debug Workflow 3 ขั้น** ข้างบน แก้บัคให้ได้
4. Commit + Push ผลงาน

### 7.5 ข้อควรระวัง
- **อย่ายัด error อย่างเดียวให้ AI** — ต้องบอกบริบทด้วยว่าโค้ดที่เกี่ยวข้องคืออะไร (paste โค้ดมาด้วย)
- **อย่าเชื่อ AI 100%** — บางครั้ง AI แก้ผิดทาง ลองเปิด Console อีกรอบหลังแก้ดูว่ามี error อื่นโผล่มาไหม
- **ถ้าวนซ้ำหลายรอบไม่หาย** — เริ่มจากการอธิบาย "พฤติกรรมที่คาดหวัง vs ที่เกิดขึ้นจริง" ให้ AI ฟังก่อน

## 💡 เทคนิคสำหรับผู้สอน (Instructor Notes)
1. **ขายความเป็น AI-First:** อธิบายว่าทำไมคาบนี้เริ่มจาก Stitch (AI สร้าง UI) ก่อนสอน CSS — เพราะหลักการของคอร์สคือ "ให้ AI ทำให้สวยก่อน แล้วค่อยเรียนรู้โค้ดทีหลัง"
2. **ห้ามจมกับ syntax:** ถ้านิสิตถามว่า `flex` เขียนยังไง ให้ตอบ *"ไม่ต้องจำ — copy ไปถาม AI ว่ามันทำอะไร แค่นั้นพอ"*
3. **เน้น Debugging เป็นทักษะหลัก:** AI-First Architect ไม่ใช่คนที่โค้ดเก่ง แต่คือคน "Debug เก่ง" ด้วย AI — ฝึกซ้ำๆ จนกลายเป็นนิสัย
4. **Design System ก่อน Visual:** อธิบายว่าทำไมคุยเรื่อง "ระบบ" ก่อน "ความสวย" — เพราะระบบทำให้ความสวยกลายเป็น scale ได้
