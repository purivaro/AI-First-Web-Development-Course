## 🎯 ส่วนที่ 1 — เปิดคาบ & แนวคิด "AI-First Web Architect"
*(15 นาที)*

### 1.1 แนวคิดนี้ต่างจากการเรียนเขียนโปรแกรมแบบเดิมอย่างไร?
ในอดีต หากคุณอยากสร้างเว็บ คุณต้องเริ่มต้นจากการท่องจำคำสั่ง (Syntax) เช่น การจำว่าพิมพ์ `<br>` คือการขึ้นบรรทัดใหม่ การสร้างหน้าเว็บต้องพิมพ์โค้ดทีละบรรทัด

**แต่ในยุค AI-First ทุกอย่างเปลี่ยนไปแล้ว:**
- **AI คือ "ผู้รับเหมาก่อสร้าง"**: AI สามารถเขียนโค้ดได้เก่งและเร็วกว่ามนุษย์ทั่วไปหลายสิบเท่า
- **คุณคือ "สถาปนิก (Architect)"**: หน้าที่ของคุณไม่ใช่การไปก่ออิฐทีละก้อน แต่คือ **การออกแบบ วางสเปก (Spec) เชื่อมต่อเครื่องมือ (Tools) และสั่งการ**
- **ทักษะที่สำคัญที่สุดเปลี่ยนไป**: จาก "การจำ Syntax โค้ด" กลายเป็น "การร่างความต้องการ (Requirements) ด้วย Markdown" และ "การใช้ AI อัตโนมัติในงานซ้ำซาก (Git)"

### 1.2 เครื่องมือ (Tools) ที่เราจะใช้เป็น "ทีมงาน" ของเรา
1. **AI Assistant / Editor (เช่น Antigravity / Cursor / Claude)**: ทำหน้าที่เป็นมือขวาในการเขียนโค้ด
2. **Markdown (`.md`)**: ภาษากลางในการสื่อสารกับ AI แบบมีโครงสร้าง
3. **Git & GitHub**: ระบบเซฟเกมและพื้นที่โชว์ผลงาน (Hosting) สำหรับมืออาชีพ

## 🌐 ส่วนที่ 2 — เว็บไซต์ทำงานยังไง? (Client-Server + HTML เบื้องต้น)
*(20 นาที)*

> 🎯 **เป้าหมาย:** ก่อนจะให้ AI สร้างเว็บให้เรา เราต้องรู้ก่อนว่า "เว็บ" จริงๆ แล้วคืออะไร ทำงานยังไง ประกอบด้วยอะไรบ้าง — เพื่อให้เราสื่อสารกับ AI ได้รู้เรื่อง และอ่านโค้ดที่ AI เขียนมาออก

### 2.1 เว็บไซต์ทำงานยังไง? (Client-Server Model)
ลองนึกภาพการ "สั่งอาหารที่ร้าน":
- **Client (ลูกค้า)** = เบราว์เซอร์ของเรา (Chrome, Safari, Edge) → เป็นคนสั่ง
- **Server (ครัว)** = คอมพิวเตอร์ที่เก็บไฟล์เว็บไว้บนอินเทอร์เน็ต → เป็นคนทำ
- **Request (การสั่ง)** = พิมพ์ URL `https://example.com` แล้วกด Enter
- **Response (อาหาร)** = Server ส่งไฟล์ HTML/CSS/JS กลับมา เบราว์เซอร์เอามาแสดงผลให้เห็น

> 💡 **สรุปสั้นๆ:** ทุกครั้งที่เปิดเว็บ = เบราว์เซอร์ขอไฟล์จาก Server → Server ส่งให้ → เบราว์เซอร์วาดออกมาให้ดู

### 2.2 Server มีกี่แบบ?
ในระดับเริ่มต้น แยกง่ายๆ เป็น 2 ประเภท:

| ประเภท | คืออะไร | เหมาะกับ | ตัวอย่างบริการ |
|---|---|---|---|
| **Static Server** | เก็บไฟล์ HTML/CSS/JS สำเร็จรูปไว้ ใครขอก็ส่งไฟล์เดิมๆ | เว็บแนะนำตัว, Portfolio, Landing Page, Blog | **GitHub Pages**, Netlify, Vercel, Cloudflare Pages |
| **Dynamic Server** | ประมวลผลก่อนส่ง เช่น ดึงข้อมูลจากฐานข้อมูลมาประกอบหน้าใหม่ทุกครั้ง | เว็บที่มี Login, ระบบโพสต์, E-commerce | Node.js, Python (Django/Flask), PHP |

> 📌 **ในคาบนี้เราจะใช้ Static Server (GitHub Pages)** เพราะฟรี ง่าย และเหมาะกับเว็บแนะนำตัวที่เราจะสร้างกัน
> ส่วน Dynamic Server (ที่มีฐานข้อมูล) เราจะเรียนใน **Session 3**

### 2.3 รู้จัก HTML — โครงกระดูกของหน้าเว็บ
**HTML (HyperText Markup Language)** คือภาษาที่ใช้บอกเบราว์เซอร์ว่า "อะไรอยู่ตรงไหน" บนหน้าเว็บ
- เปรียบเทียบ: HTML = โครงกระดูก, CSS = ผิวหนัง/เสื้อผ้า (ความสวย), JavaScript = กล้ามเนื้อ (การเคลื่อนไหว/โต้ตอบ)

**โครงสร้างพื้นฐานของไฟล์ `.html`:**
```html
<!DOCTYPE html>
<html>
  <head>
    <title>ชื่อเว็บ (แสดงบนแท็บเบราว์เซอร์)</title>
  </head>
  <body>
    <h1>หัวข้อใหญ่</h1>
    <p>ย่อหน้าข้อความ</p>
  </body>
</html>
```

**Tag ที่เจอบ่อยและควรรู้จัก:**
| Tag | หน้าที่ |
|---|---|
| `<h1>` – `<h6>` | หัวข้อ (ใหญ่ → เล็ก) |
| `<p>` | ย่อหน้าข้อความ |
| `<a href="...">` | ลิงก์ไปหน้าอื่น |
| `<img src="..." alt="...">` | รูปภาพ |
| `<ul>` / `<li>` | รายการ (Bullet list) |
| `<div>` | กล่องสำหรับจัดกลุ่ม (เหมือนตู้ใส่ของ) |
| `<button>` | ปุ่มกดได้ |

> 💡 **ไม่ต้องท่องจำ!** หน้าที่ของเราคือ "อ่านออก" พอ AI สร้างโค้ดมา เราต้องดูได้ว่ามันคืออะไรอยู่ตรงไหน — ส่วนการพิมพ์โค้ด ให้ AI ทำ

### 2.4 Workshop เล็กๆ: ส่อง HTML ของเว็บจริง
1. เปิดเว็บอะไรก็ได้ (เช่น google.com)
2. คลิกขวา → **View Page Source** (หรือ `Ctrl+U` / `Cmd+Option+U`)
3. ดูว่า "หน้าเว็บที่เราเห็น" จริงๆ แล้วประกอบจาก tag HTML แบบที่เราเพิ่งเรียน
4. ลองหา `<h1>`, `<a>`, `<img>` ในนั้นว่าอยู่ตรงไหน

> 🎯 **Key Takeaway:** ทุกเว็บบนโลกนี้ = HTML + CSS + JS ส่งจาก Server มาให้เบราว์เซอร์วาด — ไม่ว่าจะใหญ่แค่ไหน หลักการเดียวกันหมด

## 🤖 ส่วนที่ 3 — Mindset เปลี่ยนชีวิต: ใช้ AI แบบ Agent ไม่ใช่ Chatbot
*(25 นาที)*

### 3.1 ปัญหาของการใช้ AI แบบ Chatbot
นิสิตส่วนใหญ่เริ่มต้นด้วยการ "พิมพ์สั้นๆ แล้วหวังผลใหญ่" เช่น:
- *"ช่วยเขียนบทความหน่อย"*
- *"สรุปให้หน่อย"*
- *"คิดไอเดียให้หน่อย"*
- *"ทำพรีเซนต์ให้หน่อย"*

**ผลที่ได้:** ผลลัพธ์ทั่วไป คลุมเครือ ต้องแก้เยอะ เพราะ AI ไม่รู้ Context, ไม่รู้กลุ่มเป้าหมาย, ไม่รู้ข้อจำกัด

### 3.2 ทางออก: ใช้ AI แบบ Agent
แทนที่จะสั่งสั้นๆ ให้ brief งานแบบมืออาชีพ คือให้ข้อมูลครบทั้ง:
- **Context** — กลุ่มเป้าหมาย, สถานการณ์, ข้อมูลพื้นฐาน
- **Goal** — เป้าหมายชัดเจน, ผลลัพธ์ที่ต้องการ
- **Constraints** — ข้อจำกัดเวลา, รูปแบบ, สิ่งที่ต้องหลีกเลี่ยง

ผลที่ได้: ผลลัพธ์ตรงเป้า ใช้งานได้จริง ไม่ต้องแก้เยอะ

> 📌 Framework เต็มรูปแบบ (Context / Goal / Constraints / Files / Workflow) เราจะลงลึกใน Session 4

### 3.3 ศิลปะการตั้งคำถามกลับ (Clarifying Questions)
เคล็ดลับสำคัญ: **ก่อนให้ AI ทำงาน ให้ AI ถามกลับเราก่อน**

ตัวอย่าง Prompt:
> *"ฉันอยากทำเว็บแนะนำตัว แต่อย่าเพิ่งเขียนเลย ขอให้คุณถามฉันก่อนสัก 5 คำถาม เพื่อให้คุณเข้าใจกลุ่มเป้าหมายและสไตล์ที่ฉันต้องการก่อนเริ่มงาน"*

AI จะถามกลับมา เช่น:
- กลุ่มเป้าหมายคือใคร?
- ต้องการนำไปใช้แบบไหน?
- มีตัวอย่างที่ชอบไหม?
- โทนสีและความรู้สึกที่อยากได้?

เมื่อตอบครบ AI จะมีข้อมูลพอที่จะสร้างงานตรงเป้า

### 3.4 Workshop: เปรียบเทียบ Chatbot vs Agent
1. เปิด AI แล้วลองสั่งงานเดียวกันสองแบบ:
   - **แบบ Chatbot:** *"ช่วยเขียนบทความเรื่อง AI หน่อย"*
   - **แบบ Agent:** *"ฉันอยากเขียนบทความเรื่อง AI สำหรับนิสิตปี 1 ที่ไม่มีพื้นฐาน ให้อ่านง่าย ความยาว 500 คำ โทนเป็นกันเอง ห้ามใช้ศัพท์เทคนิคหนัก ก่อนเขียน ให้ถามฉันก่อน 3 คำถามว่ายังขาดข้อมูลอะไร"*
2. เปรียบเทียบผลลัพธ์ที่ได้ — เห็นความต่างชัดเจน

## 📝 ส่วนที่ 4 — Markdown ภาษากลางในการทำ "แบบแปลน" (Requirements)
*(35 นาที)*

### 4.1 ทำไมเรื่องนี้ถึงสำคัญมากในการสั่ง AI?
บริษัทเทคโนโลยีค้นพบว่า ปัญหาใหญ่ของการใช้ AI คือ **"อาการหลงลืมและหลงทาง (Context Drift)"** หากสั่งงานยาวๆ ในแชท AI จะเริ่มลืม

**🎯 ทางออก: เทคนิค Spec-Driven Development**
เราจะไม่สั่งให้ AI สร้างเว็บแบบลอยๆ แต่เราจะสร้างไฟล์ `.md` (เช่น `requirements.md`) เพื่อเป็น **Single Source of Truth (คัมภีร์หลัก)** ให้ AI อ่านเสมอ

### 4.2 ไวยากรณ์ (Syntax) เบื้องต้น
- ใช้ `# ` สำหรับหัวข้อหลัก
- ใช้ `- ` สำหรับรายการ (Bullet)
- ใช้ `**` คล่อมข้อความเพื่อทำตัวหนา

### 4.3 Workshop: ร่างแบบแปลนของฉัน
1. เปิด AI Editor ขึ้นมา
2. สร้างโฟลเดอร์โปรเจกต์ และสร้างไฟล์ `requirements.md`
3. **ใช้เทคนิค Agent:** สั่ง AI ให้ถามกลับเราก่อนเขียน
   > *"ฉันอยากทำเว็บแนะนำตัวสำหรับโชว์ผลงาน ก่อนช่วยฉันร่าง requirements.md ขอให้คุณถามฉันสัก 5 คำถามเพื่อให้เข้าใจเป้าหมายและสไตล์ที่ฉันต้องการ"*
4. ตอบคำถาม AI แล้วให้ AI สรุปเป็น `requirements.md` ที่มี: หัวข้อ, กลุ่มเป้าหมาย, โทนสี, เนื้อหาที่จะใส่, ฟีเจอร์ที่อยากได้

## 🤖 ส่วนที่ 5 — ลงมือสร้าง: ให้ AI เนรมิตโค้ด
*(25 นาที)*

เมื่อเรามี **แบบแปลน (`requirements.md`)** ที่ครบถ้วน ถึงเวลาสั่งผู้รับเหมาลงมือ!

### 5.1 คำสั่งเดียว (The Master Prompt)
เปิดแชทใน AI Editor แล้วพิมพ์คำสั่ง:
> *"ช่วยอ่านไฟล์ `requirements.md` แล้วเขียนเว็บไซต์ตามนั้นให้สมบูรณ์ (HTML/CSS/JS) ใช้โทนสีและสไตล์ตามที่ระบุไว้ ถ้ามีอะไรที่ไม่ชัดเจน ให้ถามฉันก่อน"*

### 5.2 AI ทำงาน (Magic Moment)
AI จะประมวลผล:
1. อ่านเนื้อหาที่คุณร่างไว้
2. ถามกลับถ้ามีจุดไม่ชัด (ตามหลัก Agent mindset)
3. เขียนไฟล์โค้ด (เช่น `index.html` หรือ `App.jsx`) ขึ้นมาให้ทันที
- **ผลลัพธ์:** คุณจะได้เว็บที่ตรงตามแบบแปลน โดยไม่ต้องพิมพ์โค้ดเอง!

## 🚀 ส่วนที่ 6 — Version Control & Deploy: เอาขึ้นออนไลน์แบบเหนือชั้น
*(25 นาที)*

ตอนนี้เว็บยังอยู่ในเครื่องเรา เราต้องส่งขึ้น "GitHub" เพื่อให้คนทั่วโลกเข้าถึงได้

### 6.1 เข้าใจ Git ใน 1 นาที
การทำงานกับ Git เหมือนการแพ็คของส่งไปรษณีย์:
- `git add .` = เอาของทั้งหมดใส่กล่อง
- `git commit` = ปิดกล่องและเขียนป้ายกำกับ (บอกว่าอัปเดตอะไร)
- `git push` = ส่งกล่องขึ้นรถไปรษณีย์ (GitHub)

### 6.2 ให้ AI พิมพ์คำสั่ง Git ให้ (AI-Assisted Workflow)
แทนที่คุณจะต้องเปิดหน้าจอดำ (Terminal) มาพิมพ์คำสั่งเองทีละบรรทัด เราสามารถใช้พลังของ AI Automation ได้!
- **คำสั่งที่คุณพิมพ์บอก AI ในแชท:**
  > *"งานส่วนนี้เสร็จแล้ว ช่วยทำการ `git add .`, `git commit` ด้วยข้อความว่า 'สร้างหน้าเว็บแรก', และ `git push` โค้ดทั้งหมดขึ้น GitHub ให้ฉันที"*
- AI จะรันคำสั่งเหล่านี้เบื้องหลังให้อัตโนมัติ (อาจมี popup ถามยืนยันเพื่อความปลอดภัย) นี่คือประสบการณ์ของสถาปนิกตัวจริงที่ให้ลูกน้องจัดการงานน่าเบื่อ!

### 6.3 เปิดใช้งาน GitHub Pages
เมื่อโค้ดไปอยู่บน GitHub แล้ว ขั้นตอนสุดท้ายคือการเปิดสวิตช์ให้เป็นเว็บไซต์:
1. ไปที่หน้า GitHub Repository ของคุณ
2. ไปที่ **Settings > Pages**
3. หัวข้อ Build and deployment ตรง Branch ให้เลือก `main` (หรือ branch หลักของคุณ)
4. กด Save และรอประมาณ 1 นาที
5. รีเฟรชหน้าเว็บ คุณจะได้ URL ของจริง เช่น `https://ชื่อคุณ.github.io/my-website` เอาไปอวดเพื่อนได้เลย!
